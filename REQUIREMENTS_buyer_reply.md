## 1. ゴール

eBay でバイヤーから来たメッセージに対し、AI が返信案を生成し、青田さんが Discord 上の
ボタンでワンクリック送信できるようにする。送信ログはスプレッドシートに残す。

---

## 2. システム構成

```
[eBay Messages]
   ↓ (1) 新着メッセージ取得（ポーリング）
[Backend]
   ↓ (2) AI返信案生成（過去履歴を参考）
[Discord 通知]（メッセージ + ボタン）
   ↓ (3) ボタン押下: 送信 / 編集 / オファー受諾 / オファー拒否
[Backend]（Discord Interaction を受信）
   ↓ (4) eBay API で送信
[Google Spreadsheet]
   ↑ (5) 在庫発送_作業 タブに返信ログ記録
```

---

## 3. 機能要件

### F1: メッセージ受信
- eBay Messages API（Trading API の `GetMyMessages`）で新着メッセージを取得
- **【補完】** ポーリング間隔: 5分（前半システムと統一）
- **【補完】** 重複検知キー: メッセージID（`MessageID`）。処理済IDはスキップ
- **【補完】** 初回取得範囲: 過去3日分

### F2: AI 返信案生成
- バイヤーのメッセージ + 過去のやり取り履歴を元に Claude API で返信文を生成
- **【補完】返信言語**: バイヤーのメッセージ言語に合わせる（英語メッセージ→英語返信が基本）
- **【補完】トーン**: 丁寧・簡潔・eBayセラーとして適切な範囲。青田さんの過去返信をfew-shotで反映
- **【補完】参照する履歴**: 同一バイヤー・同一注文の過去メッセージ（スプシ or eBay側履歴）
- **【補完】生成失敗時**: 「返信案生成エラー」と Discord 表示し、手動入力を促す
- **【補完】人間レビュー必須**: AI返信は必ず青田さんの確認を経て送信（自動送信はしない）

### F3: Discord 通知
- 「誰から / 何が来た / AI返信案」を Discord に投稿し、操作ボタンを付ける
- **【重要・補完】技術的訂正**:
  - loose 仕様は「Discord Webhook + Interactions」とあるが、**Webhook 単体ではボタン操作（Interaction）を受信できない**
  - ボタン押下を処理するには **Discord Application（Bot）+ Interactions Endpoint URL（HTTPSで公開された受信エンドポイント）** が必要
  - もしくは Discord Bot をホストして Gateway 接続する方式
  - → **本システムは Discord Bot 方式が必須**（後述アーキテクチャ参照）

### F4: ワンクリック送信
- Discord のボタンを押すと eBay にメッセージ送信
- **【補完】ボタン種別**: 「送信」「編集して送信」「オファー受諾」「オファー拒否」
- **【補完】「編集」フロー**: Discord Modal で返信文を編集 → 送信
- **【補完】二重送信防止**: 送信済メッセージのボタンは無効化（disabled）に更新
- eBay 送信 API: Trading API `AddMemberMessageRTQ`（取引相手への返信）

### F5: オファー処理
- 値下げオファー受信時に「受諾 / 拒否」ボタンを表示
- eBay API: Trading API `RespondToBestOffer`（受諾/拒否/カウンター）
- **【補完】カウンターオファー**: v1 ではスコープ外（受諾/拒否のみ）

### F6: スプシ記録
- 送信後、対象オーダー行に返信内容を記録
- シート: `在庫発送_作業`、キー: `Order no.`（eBay Order ID で紐付け）
- 追加列: `最終バイヤーメッセージ` / `AI返信案` / `送信した返信文` / `送信日時` / `オファー判断`（受諾/拒否/なし）
- **【補完】** Order no. が紐付かないメッセージ（取引前の質問）は別タブ `返信ログ` に記録
- **【補完】`返信ログ` タブ列設計**: メッセージID / 受信日時 / バイヤーID / 件名 / 本文 / AI返信案 / 送信文 / 送信日時

## 4. 【補完】非機能要件

| 区分 | 要件 |
|---|---|
| レスポンス | 新着検知から Discord 通知まで 5分以内（ポーリング間隔依存） |
| 返信生成時間 | Claude API 1回 数秒以内 |
| 可用性 | バックエンドは常時稼働（ホスティング必須、後述） |
| セキュリティ | eBay/Discord/Claude/Google の各キーは環境変数 or Secret Manager 管理。コード直書き禁止 |
| Discord検証 | Interactions Endpoint は Ed25519 署名検証必須（Discord仕様） |
| データ保持 | 返信ログは無期限（業務記録）。要クライアント確認 |
| 監査 | 送信操作（誰がいつ何を送ったか）をログ化 |

## 5. 【補完】アーキテクチャ（技術スタック確定案）

loose 仕様の「Python/Node + Cloud Run/Vercel/Render（要検討）」を以下で確定提案:

### 推奨案A: Python (FastAPI) + Cloud Run
- **Backend**: FastAPI（Python 3.11+）
- **ポーリング**: Cloud Scheduler → Cloud Run（5分間隔で eBay Messages 取得）
- **Discord Interactions 受信**: Cloud Run の HTTPS エンドポイント（Ed25519 署名検証）
- **AI**: Claude API（Anthropic）
- **記録**: Google Sheets API（Service Account）
- **シークレット**: Google Secret Manager
- **理由**: Cloud Scheduler でポーリング、常時HTTPSエンドポイント、Secret Manager が一体で揃う

### 代替案B: GAS で最速デモ（5/21提示向け）
- **メリット**: 前半システムの eBay OAuth / Sheets / Discord 投稿コードを流用でき、デモを最速で出せる
- **制約**: Discord Interactions の Ed25519 署名検証が GAS では実装が重い。ボタン送信は GAS doPost で受けられるが署名検証回りが鬼門
- **位置付け**: 「5/21に動くものを見せる」だけなら、F1〜F3（取得→AI生成→Discord通知）まで GAS で作り、F4送信ボタンは簡易リンク方式で代替するデモも可能

> **判断ポイント**: 5/21デモを優先するなら案B簡易版、本番品質なら案A。→ §8の「決めるべきこと」参照
